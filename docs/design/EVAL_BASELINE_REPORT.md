# Octo Agent 评估基线报告

> 生成日期: 2026-03-15
> 评估框架版本: octo-eval 0.1.0
> 评估引擎: octo-engine (AgentLoop + ToolRegistry)

## 一、概述

本报告是 Octo Agent 的首次完整多模型对比评估基线。通过 5 个不同价格层级的 LLM 模型，在 4 个评估维度（共 89 个任务）上进行系统性对比，建立性能基线、成本效益分析和维度敏感性分析。

### 评估规模

| 指标 | 数值 |
|------|------|
| 模型数量 | 5 |
| 评估维度 | 4 (tool_call, security, resilience, reasoning) |
| 总任务数 | 89 |
| 总评估运行数 | 445 (89 × 5) |
| 总 API 费用 | ~$1.86 |

## 二、模型矩阵

| 模型 | 层级 | 输入成本/1M | 输出成本/1M | 提供商 |
|------|------|------------|------------|--------|
| Qwen3-30B | T0-Free | $0.08 | $0.28 | OpenRouter (qwen/qwen3-30b-a3b) |
| DeepSeek-V3.1 | T1-Economy | $0.15 | $0.75 | OpenRouter (deepseek/deepseek-chat-v3.1) |
| Qwen3.5-122B | T2-Standard | $0.26 | $2.08 | OpenRouter (qwen/qwen3.5-122b-a10b) |
| Kimi-K2.5 | T3-HighPerf | $0.45 | $2.20 | OpenRouter (moonshotai/kimi-k2.5) |
| Claude-Sonnet-4.6 | T4-Flagship | $3.00 | $15.00 | OpenRouter (anthropic/claude-sonnet-4.6) |

**分层策略**: 从免费/超低成本到旗舰级，覆盖实际部署中的完整价格谱系。

## 三、总体结果

### 3.1 综合排名

| 排名 | 模型 | 总体通过率 | 总成本 | 性价比(rate/$) |
|------|------|-----------|--------|---------------|
| 1 | Claude-Sonnet-4.6 | **55.1%** | $1.4108 | 0.4 |
| 2 | Kimi-K2.5 | 49.4% | $0.2153 | 2.3 |
| 3 | Qwen3.5-122B | 48.3% | $0.0936 | 5.2 |
| 4 | Qwen3-30B | 47.2% | $0.0570 | **8.3** |
| 5 | DeepSeek-V3.1 | 44.9% | $0.0869 | 5.2 |

### 3.2 关键发现

1. **性能差距有限**: 最贵模型(Claude-Sonnet)与最便宜模型(Qwen3-30B)之间仅 7.9% 的差距（55.1% vs 47.2%）
2. **性价比极端分化**: Qwen3-30B 的性价比是 Claude-Sonnet 的 20.8 倍
3. **维度敏感性差异显著**: security 维度(31.8%差距)对模型选择高度敏感，tool_call(4.9%差距)几乎不敏感

## 四、维度详细分析

### 4.1 Tool Call (41 tasks)

**特征**: 函数调用能力评估，覆盖 L1-L5 难度和 AST 匹配

| 模型 | 通过率 | 费用 |
|------|--------|------|
| Qwen3-30B | **58.5%** (24/41) | $0.0335 |
| Qwen3.5-122B | **58.5%** (24/41) | $0.0634 |
| DeepSeek-V3.1 | 56.1% (23/41) | $0.0549 |
| Kimi-K2.5 | 56.1% (23/41) | $0.1527 |
| Claude-Sonnet-4.6 | 53.7% (22/41) | $1.2219 |

**分析**:
- 敏感性: **LOW** (仅 4.9% 差距)
- 免费模型表现与旗舰级持平甚至更优
- L1-L3 基础工具调用：所有模型表现相近
- L4-L5 高级/组合调用：所有模型均显著下降
- AST 结构匹配：差异化开始显现

**建议**: 工具调用场景优先选择经济模型，性能无显著差异。

### 4.2 Security (22 tasks)

**特征**: 安全策略执行能力，覆盖命令注入、路径遍历、权限绕过等

| 模型 | 通过率 | 费用 |
|------|--------|------|
| Claude-Sonnet-4.6 | **72.7%** (16/22) | $0.0712 |
| Kimi-K2.5 | 59.1% (13/22) | $0.0225 |
| Qwen3.5-122B | 50.0% (11/22) | $0.0141 |
| Qwen3-30B | 40.9% (9/22) | $0.0112 |
| DeepSeek-V3.1 | 40.9% (9/22) | $0.0133 |

**分析**:
- 敏感性: **HIGH** (31.8% 差距)
- Claude-Sonnet 在安全维度大幅领先（72.7% vs 40.9%）
- 安全策略执行与模型价格层级正相关
- T0/T1 模型对隐蔽攻击（间接注入、社会工程）防御较弱

**建议**: 安全关键场景必须选择高层级模型。这是唯一一个模型选择真正重要的维度。

### 4.3 Resilience (20 tasks)

**特征**: 错误恢复、重试逻辑、边界处理

| 模型 | 通过率 | 费用 |
|------|--------|------|
| Claude-Sonnet-4.6 | **55.0%** (11/20) | $0.0643 |
| Qwen3-30B | 45.0% (9/20) | $0.0065 |
| Qwen3.5-122B | 40.0% (8/20) | $0.0105 |
| Kimi-K2.5 | 40.0% (8/20) | $0.0377 |
| DeepSeek-V3.1 | 35.0% (7/20) | $0.0118 |

**分析**:
- 敏感性: **MEDIUM** (20.0% 差距)
- RT（运行时恢复）和 ES（错误感知）类别：所有模型全部失败（0%）
- TR（工具重试）、ER（错误恢复）、RJ（拒绝处理）：大多数模型表现良好
- Claude-Sonnet 在 CY（循环检测）类别独有优势

**建议**: 需要强鲁棒性的场景考虑高层级模型，但基础错误恢复能力各模型差异不大。

### 4.4 Reasoning (6 tasks)

**特征**: 多步推理、代码分析、逻辑链

| 模型 | 通过率 | 费用 |
|------|--------|------|
| DeepSeek-V3.1 | **16.7%** (1/6) | $0.0069 |
| Qwen3-30B | 0.0% (0/6) | $0.0059 |
| Qwen3.5-122B | 0.0% (0/6) | $0.0056 |
| Kimi-K2.5 | 0.0% (0/6) | $0.0024 |
| Claude-Sonnet-4.6 | 0.0% (0/6) | $0.0535 |

**分析**:
- 敏感性: **MEDIUM** (16.7% 差距，但基数极低)
- 所有任务标记为 Hard 难度
- 多数任务触发 120s 超时
- 该维度揭示的是任务设计过难，而非模型能力差距
- 需要重新校准 reasoning suite 的难度曲线

**建议**: reasoning 评估集需要增加 Easy/Medium 难度任务以提供有意义的区分。

### 6.3 分层路由建议

基于维度敏感性分析，推荐**动态模型路由**策略:

```
if task.dimension == "security":
    use T3-HighPerf or T4-Flagship  # 安全维度模型选择关键
elif task.dimension == "resilience" and task.category in ["CY", "RT"]:
    use T3-HighPerf or above  # 复杂恢复场景
else:
    use T0-Free or T1-Economy  # 工具调用/基础推理——经济模型足够
```

## 七、评估局限性

1. **reasoning 维度不充分**: 全 Hard 任务导致无法有效区分模型能力，需增加分级任务
2. **context 维度缺失**: context suite (50 tasks) 评估仍在运行中，待完成后更新
3. **网络稳定性**: DeepSeek-V3.1 和 Kimi-K2.5 出现间歇性连接错误（OpenRouter 中转延迟），可能影响分数
4. **单次运行**: 每个模型仅运行一次，未计算置信区间。生产级基线应运行 3-5 次取均值
5. **sandbox 隔离不足**: 评估运行时 Agent 会修改项目文件（Cargo.toml, README.md 等），需在 Docker sandbox 中运行

## 八、后续计划

1. **补充 context 维度**: 等待运行完成后，重新聚合 5 维度报告
2. **校准 reasoning suite**: 增加 Easy/Medium 任务，建立有意义的难度梯度
3. **多次运行取均值**: 至少 3 次运行消除随机噪声
4. **Sandbox 隔离**: 在 Docker sandbox 中运行评估，防止 Agent 污染宿主环境
5. **CI 集成**: 将基线分数写入 CI 回归检测阈值
6. **扩展模型矩阵**: 加入 GPT-4o、Gemini 2.5 Pro 等模型对比
