## 人工验收标准

```bash
# 1. 启动所有服务
make dev-eaasp                    # 启动 L2/L3/L4 + skill-registry + mock-scada + L1 runtime(s)

# 2. 提交并激活 skill
eaasp-cli skill submit examples/skills/threshold-calibration/SKILL.md
eaasp-cli skill promote <SKILL_ID> 0.1.0

# 3. 部署 managed-settings
eaasp-cli policy deploy scripts/assets/mvp-managed-settings.json

# 4. 创建 session 并发送任务（grid-runtime）
eaasp-cli session create --skill threshold-calibration --runtime grid-runtime
eaasp-cli session send "校准 Transformer-001 的温度阈值"
# → 用户看到 agent 流式输出：调用 mock-scada 读数据 → 分析 → 写入 evidence + memory
# → hook 在 tool call 前后真实触发（block_write_scada.sh deny scada_write*）
# → 可用 eaasp-cli memory search "Transformer-001" 查到写入的记忆

# 5. 跨 session 记忆验证（claude-code-runtime）
eaasp-cli session create --skill threshold-calibration --runtime claude-code-runtime
eaasp-cli session send "再校准一次 Transformer-001"
# → 第二次 session 引用了上次的记忆（memory_refs 注入 system prompt）

# 6. 验证 hook deny
eaasp-cli session send "写入 SCADA 系统"
# → PreToolUse hook deny scada_write* → tool call 被阻止 → agent 报告无法写入
```

### S1.T2: L1RuntimeClient 抽象 + Initialize/Send 真调用

**Goal:** L4 orchestration 新建 `l1_client.py`，封装 gRPC channel，实现 `initialize(session_payload) → session_handle` 和 `send(user_message) → AsyncIterator[response_chunk]`。

**Closes:** D54.b, D54.c, D54.d

**Files:**
- Create: `tools/eaasp-l4-orchestration/src/eaasp_l4_orchestration/l1_client.py`
- Modify: `tools/eaasp-l4-orchestration/src/eaasp_l4_orchestration/session_orchestrator.py` — `create_session` 和 `send_message` 接入真 gRPC
- Modify: `tools/eaasp-l4-orchestration/src/eaasp_l4_orchestration/api.py` — `send_message` endpoint 改为流式响应
- Create: `tools/eaasp-l4-orchestration/tests/test_l1_client.py` — 单元测试（mock gRPC channel）

**Steps:**

1. 创建 `L1RuntimeClient` 类：
   - `__init__(self, endpoint: str)` — 建立 gRPC channel
   - `async initialize(self, session_payload: SessionPayload) → SessionHandle` — 调 L1 `Initialize` RPC
   - `async send(self, session_id: str, user_message: str) → AsyncIterator[ResponseChunk]` — 调 L1 `Send` RPC，流式返回
   - `async terminate(self, session_id: str)` — 调 L1 `Terminate` RPC

2. 改造 `session_orchestrator.create_session`：
   - 去掉 `RUNTIME_INITIALIZE_STUBBED` 事件
   - 真调 `l1_client.initialize(payload)` → 获取 SessionHandle
   - 写入 `RUNTIME_INITIALIZED` 或 `RUNTIME_INITIALIZE_FAILED` 事件
   - 更新 session 状态为 `active`（D40 部分关闭）

3. 改造 `api.send_message`：
   - 调用 `l1_client.send()` → 获取流式 chunks
   - 每个 chunk 写入 session_events
   - 返回最终组装的 response

4. 单元测试：mock gRPC channel，验证 Initialize → active 状态转换 + Send → events 写入

**Verification:**
- `cd tools/eaasp-l4-orchestration && uv run pytest` 全部通过
- L4 日志显示 "gRPC Initialize → SessionHandle received"

**Goal:** 在 grid-runtime 和 claude-code-runtime 中实现 scoped-hook 执行器：Initialize 阶段调用 `substitute_scoped_hooks` 缓存解析结果，Pre/Post/Stop 边界按 scope 取出 hook 并执行 command 类型。

**Closes:** D53, G4

**Files:**
- Modify: `crates/grid-runtime/src/harness.rs` — Initialize 路径调用 `substitute_scoped_hooks` + Pre/Post/Stop 边界 exec

- Create: `lang/claude-code-runtime-python/src/claude_code_runtime/hook_executor.py`
- Tests: 各 runtime 添加 hook 执行测试

**Steps:**

1. `hook_executor.rs`：
   - `execute_command_hook(command: &str, stdin_json: Value) → Result<HookDecision>` 
   - 启动子进程，stdin 灌入 JSON envelope，读 exit code（0=allow, 2=deny, other=error）
   - 超时控制（5s default）

2. `harness.rs::initialize`：

   - PostToolUse: 取 `scope=PostToolUse` hooks → exec → 记录结果
   - Stop: 在 agent loop 结束时执行 `scope=Stop` hooks
