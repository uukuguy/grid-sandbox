## 第一部分：项目概述与优先级

### 1.1 三个 Deferred 项

| 优先级 | ID | 功能 | 复杂度 | 预计任务数 | 说明 |
|--------|-----|------|--------|-----------|------|
| P0 | D5 | 双 Agent 模式 (Plan + Build) | 中 | 6 | 核心交互差异化，直接增强 REPL 体验 |
| P1 | D4 | Auto-memory 跨 Session | 中 | 6 | 自动学习项目记忆，提升连续性 |
| P2 | D2 | 嵌入式 Web Dashboard | 高 | 8 | Alpine.js SPA，`include_str!()` 单二进制 |

### 1.2 优先级决策理由

1. **D5 (双 Agent)** 排最高：A1 已实现 `/mode plan|build` 切换，但目前只是切换 system prompt 前缀，缺少真正的 Tab 式双 Agent 并行工作流。这是 octo-cli 相对竞品的核心差异化功能。
2. **D4 (Auto-memory)** 排第二：跨 Session 记忆直接影响用户体验连续性，且 engine 已有 `MemorySystem`/`SemanticMemory`/`KnowledgeGraph` 基础设施。
3. **D2 (Web Dashboard)** 排最后：依赖 TUI backend trait 已完成，但工作量最大（Alpine.js + 嵌入 HTML），且 TUI 已提供良好的全屏体验。

## 第二部分：Phase 6 — 双 Agent 模式 (D5)

### 2.1 目标

实现 Plan Agent 与 Build Agent 的真正分离：
- Plan Agent：只分析、规划，不执行工具调用
- Build Agent：执行代码编写、文件操作、命令运行
- REPL 中可用 `/switch` 或 Tab 键切换
- Plan 的输出可自动导入 Build 的上下文

### 2.2 任务分解

| ID | 任务 | 依赖 | 说明 |
|----|------|------|------|
| D5-1 | AgentProfile 数据结构 | — | 定义 `AgentProfile { name, system_prompt, tool_filter, mode }` |
| D5-2 | DualAgentManager | D5-1 | 管理 Plan/Build 两个 AgentExecutor 实例，共享 Session |
| D5-3 | REPL 集成 — `/switch` 命令 | D5-2 | 在 REPL 中切换活跃 Agent，状态栏显示当前 Agent |
| D5-4 | Plan → Build 上下文传递 | D5-2 | Plan Agent 输出的步骤列表自动注入 Build Agent 上下文 |
| D5-5 | TUI 双面板视图 | D5-2 | TUI 模式下左右分屏显示 Plan/Build 对话 |
| D5-6 | 测试与集成验证 | D5-3, D5-4 | 单元测试 + 集成测试 |

### 2.3 技术设计

```
DualAgentManager
├── plan_agent: AgentExecutor (tool_filter = [])  // 禁用所有工具
├── build_agent: AgentExecutor (tool_filter = ALL) // 全部工具
├── shared_session: SessionId
└── plan_context: Vec<PlanStep>  // Plan 输出缓存

ReplContext
├── active_agent: AgentSlot { Plan, Build }
├── dual_manager: Option<DualAgentManager>
└── ...existing fields...
```

**Plan Agent System Prompt 差异**:
- 不允许工具调用，只输出分析和步骤
- 输出格式约束：Markdown 步骤列表
- 自动提取步骤注入 Build Agent 上下文

## 第三部分：Phase 7 — Auto-memory 跨 Session (D4)

### 3.1 目标

Agent 自动从对话中提取关键信息（项目结构、用户偏好、常用命令），存入长期记忆层，跨 Session 复用。

### 3.2 任务分解

| ID | 任务 | 依赖 | 说明 |
|----|------|------|------|
| D4-1 | MemoryExtractor trait | — | 定义自动提取接口：从对话消息中提取 MemoryEntry |
| D4-2 | LLM-based Extractor | D4-1 | 用小模型（或规则）自动识别关键信息：文件路径、命令、偏好 |
| D4-3 | SessionEndHook 集成 | D4-1, D4-2 | Session 结束时自动触发提取，写入 MemoryStore (L2) |
| D4-4 | Session 启动记忆注入 | — | 新 Session 启动时从 L2 检索相关记忆，注入 system prompt |
| D4-5 | `/memory auto` 命令 | D4-3 | 手动触发提取 + 查看自动记忆内容 + 删除/编辑 |
| D4-6 | 测试与集成验证 | D4-3, D4-4 | 提取准确性测试、注入测试 |

### 3.3 技术设计

```
MemoryExtractor (trait)
├── extract(messages: &[ChatMessage]) -> Vec<MemoryEntry>
└── categorize(entry: &MemoryEntry) -> MemoryCategory

MemoryCategory:
├── ProjectStructure  // 文件路径、模块关系
├── UserPreference    // 编码风格、工具偏好
├── CommandPattern    // 常用命令序列
├── TechnicalDecision // 技术选型、设计决策
└── ContextualFact    // 项目特定事实

注入流程:
Session Start → L2 Search (project_id + tags) → Top-K 记忆 → SystemPrompt 追加
Session End   → Extractor(messages) → Dedup → L2 Store
```

**去重策略**：
- 基于 key 字段的精确匹配
- 基于 embedding 的语义去重 (> 0.9 相似度视为重复)
- 保留最新版本，更新 timestamp

## 第四部分：Phase 8 — 嵌入式 Web Dashboard (D2)

### 4.1 目标

在 CLI 二进制中嵌入一个轻量 Web Dashboard：
- `octo dashboard` 启动本地 HTTP 服务
- `include_str!()` 嵌入 HTML/JS/CSS，零外部依赖
- Alpine.js 作为前端框架（< 15KB gzipped）
- 复用 TuiBackend trait 的 HTTP 后端

### 4.2 任务分解

| ID | 任务 | 依赖 | 说明 |
|----|------|------|------|
| D2-1 | Dashboard 命令与 HTTP 服务器 | — | `octo dashboard [--port 8080]`，Axum 嵌入式服务器 |
| D2-2 | HTML 骨架与 Alpine.js 集成 | D2-1 | 单页面 HTML，Alpine.js CDN-free（嵌入） |
| D2-3 | Chat 面板 | D2-2 | WebSocket 流式对话，复用 AgentEvent SSE |
| D2-4 | Session 管理面板 | D2-2 | 列表/创建/删除/导出 Session |
| D2-5 | Memory 浏览面板 | D2-2 | 多层记忆浏览 (L0/L1/L2)，搜索 |
| D2-6 | MCP 管理面板 | D2-2 | MCP Server 启停、工具列表、日志 |
| D2-7 | 配色主题系统 | D2-2 | 12 种主题切换，CSS 变量（复用 TUI 主题定义） |
| D2-8 | 测试与集成验证 | D2-3 至 D2-7 | HTTP API 测试、嵌入资源测试 |

### 4.3 技术设计

```
octo-cli/src/
├── commands/dashboard.rs     // Dashboard 命令入口
├── dashboard/
│   ├── mod.rs                // 模块声明
│   ├── server.rs             // Axum HTTP 服务器
│   ├── api.rs                // REST API 路由 (复用 TuiBackend)
│   ├── ws.rs                 // WebSocket 处理
│   └── assets/
│       ├── index.html        // 主页面 (include_str!)
│       ├── app.js            // Alpine.js 应用逻辑
│       ├── alpine.min.js     // Alpine.js 运行时
│       └── style.css         // 样式 (含 12 主题)

嵌入方式:
const INDEX_HTML: &str = include_str!("assets/index.html");
const APP_JS: &str = include_str!("assets/app.js");
const STYLE_CSS: &str = include_str!("assets/style.css");
```

**与 octo-server 的区别**：
- octo-server 是独立的后端服务，含完整 REST API 和 React 前端
- Dashboard 是 CLI 嵌入的轻量版，零依赖，单二进制
- 共享 engine API（通过 TuiBackend trait），但 Dashboard 更轻量

## 第六部分：Engine API 新增需求

### Phase 6 需要的 Engine 变更

| 变更 | 模块 | 说明 |
|------|------|------|
| `AgentProfile` struct | `agent/` | Agent 配置档（name, system_prompt, tool_filter） |

### Phase 7 需要的 Engine 变更

| 变更 | 模块 | 说明 |
|------|------|------|
| `MemoryExtractor` trait | `memory/` | 自动提取接口 |

| `SessionHook::on_end()` | `session/` | Session 结束回调 |

Dashboard 完全通过 `TuiBackend` trait 访问 engine，无需新增 engine API。
