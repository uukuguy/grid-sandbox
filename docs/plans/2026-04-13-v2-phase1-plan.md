## S0. ADR 决策（Phase 1 前置，必须先完成）

### S0.T1 — ADR-V2-001: EmitEvent 接口形式 ✅

**状态**：✅ Accepted — `adrs/ADR-V2-001-emit-event-interface.md`

**目标**：决定 L1 runtime 如何将事件传递给 L4

**选项分析**：

| 选项 | 描述 | 优劣 |
|------|------|------|
| **A. MUST 方法** | EmitEvent 成为 certifier 12→13 MUST 核心方法 | 最清晰，但增加认证门槛 |
| **B. Hook-bridge 副作用** | 事件通过 hook-bridge gRPC sidecar 写入，L1 无需新接口 | 对 T2/T3 runtime 友好，但 T1 有性能开销 |
| **C. 平台拦截器** | L4 从现有 gRPC 调用（OnToolCall/OnToolResult/OnStop）提取事件 | 零 runtime 改造，但事件粒度受限 |

**建议方向**：混合方案 — EmitEvent 保持 OPTIONAL（不入 MUST），但 T1 runtime 必须实现；T2/T3 通过 hook-bridge 桥接。

**产出**：`docs/design/EAASP/adrs/ADR-V2-001-emit-event-interface.md`

**前置**：无

---

### S0.T2 — ADR-V2-002: Session Event Stream 后端选型 ✅

**状态**：✅ Accepted — `adrs/ADR-V2-002-event-stream-backend.md`

**目标**：决定事件持久化后端

**选项分析**：

| 选项 | 描述 | Phase 1 可用性 | 生产就绪 |
|------|------|---------------|---------|
| **A. SQLite WAL 增强** | 当前实现升级：WAL mode + FTS5 索引 + 分片 | 立即可用，零依赖 | 单节点可用，多节点不行 |
| **B. NATS JetStream** | 轻量级消息系统，at-least-once | 1 天搭建 | 生产级，但多一个依赖 |
| **C. Kafka** | 企业级流处理 | 2-3 天搭建 | 过重 |
| **D. S3 append-only** | 对象存储 append | 延迟高 | 归档用 |

**建议方向**：Phase 1 用 **SQLite WAL 增强**（A），接口层抽象允许后续切换。理由：
- 零外部依赖，与 Phase 0 渐进式改进一致
- dev 环境不需要额外服务
- 接口抽象保留切换能力
- 生产环境替换时只换实现不改接口

**产出**：`docs/design/EAASP/adrs/ADR-V2-002-event-stream-backend.md`

**前置**：无

### S0.T3 — ADR-V2-003: Event Clustering 插件化接口 ✅

**状态**：✅ Accepted — `adrs/ADR-V2-003-event-clustering-interface.md`

**目标**：定义 Event Engine 处理管线的 handler 接口

**v2.0 规范要求 4 种 handler type**：

| Handler | 职责 | Phase 1 实现 |
|---------|------|-------------|
| **Ingestor** | 接收原始事件，标准化格式 | 必做 |
| **Deduplicator** | 去重（相同事件在短时间窗口内合并） | 必做（简单时间窗口） |
| **Clusterer** | 按 topology/时间/因果关系聚类 | 简单实现（时间窗口聚类，无 topology ontology） |
| **Indexer** | 建立事件索引供查询 | 必做（FTS5 + time index） |

**建议方向**：定义 trait/interface，Phase 1 提供默认实现；Clusterer 推迟 topology-aware 到 Phase 5。

**产出**：`docs/design/EAASP/adrs/ADR-V2-003-event-clustering-interface.md`

**前置**：无

## S1. Event Stream 后端升级

### S1.T1 — EventStreamBackend 抽象接口 + SQLite WAL 实现

**目标**：将 `SessionEventStream` 重构为可插拔后端

**涉及文件**：
- `tools/eaasp-l4-orchestration/src/eaasp_l4_orchestration/event_stream.py` — 重构
- `tools/eaasp-l4-orchestration/src/eaasp_l4_orchestration/event_backend.py` — 新建：抽象接口
- `tools/eaasp-l4-orchestration/src/eaasp_l4_orchestration/event_backend_sqlite.py` — 新建：SQLite WAL 实现

**实现要点**：
1. 定义 `EventStreamBackend` Protocol（ABC）：
   - `append(session_id, event_type, payload, metadata) → event_id`
   - `list_events(session_id, from_seq, to_seq, limit) → list[Event]`
   - `subscribe(session_id, from_seq) → AsyncIterator[Event]` （新增：支持 follow mode）
   - `count(session_id) → int`
2. `SqliteWalBackend` 实现：
   - WAL mode 启用
   - `subscribe` 通过 polling + asyncio.Event 通知实现
   - FTS5 索引 `event_type + payload_json`
3. `SessionEventStream` 改为委托给 `EventStreamBackend`
4. 保持向后兼容：现有调用方无需改动

**测试**：6-8 个测试（append/list/subscribe/count/WAL mode/FTS search）

**前置**：S0 完成

### S1.T2 — Event 数据模型增强

**目标**：为 Event Engine 扩展事件数据模型

**涉及文件**：
- `tools/eaasp-l4-orchestration/src/eaasp_l4_orchestration/event_models.py` — 新建

**实现要点**：
1. 定义标准化事件结构（对齐 proto EventStreamEntry）：
   ```python
   @dataclass
   class Event:
       event_id: str          # UUID
       session_id: str
       event_type: str        # HookEventType 枚举值

       created_at: int        # unix epoch
   ```

2. `EventMetadata` 支持因果追踪（parent_event_id → 构建事件 DAG）
3. DB migration：扩展 `session_events` 表新增 `event_id`, `source`, `metadata_json`, `cluster_id` 列

### S2.T1 — grid-runtime EmitEvent 实装

**目标**：grid-runtime 在 agent loop 关键节点发射事件

**涉及文件**：
- `crates/grid-runtime/src/harness.rs` — 在 tool call / tool result / thinking / stop 等节点调用 EmitEvent gRPC
- `crates/grid-runtime/src/service.rs` — EmitEvent RPC handler（发给 L4）
- 可能需要 `crates/grid-runtime/src/event_emitter.rs` — 新建：事件发射器封装

**实现要点**：
1. 在 `GridHarness` 的 agent loop 中识别关键事件节点：
   - `SESSION_START` — Initialize 成功后
   - `PRE_TOOL_CALL` / `POST_TOOL_CALL` — OnToolCall/OnToolResult 前后
   - `STOP` — agent 产出 final answer
   - `THINKING_START` / `THINKING_END` — extended thinking 边界

2. 构建 `EventStreamEntry` protobuf 消息
3. 通过 gRPC 调用 L4 的事件接收端点（或通过反向 gRPC stream）
4. **非阻塞**：EmitEvent 失败不应阻止 agent 继续运行（fire-and-forget + local buffer）

**测试**：5-6 个测试（emit on tool call / emit on stop / emit failure resilience / event format）

## S3. L4 Event Engine

### S3.T1 — Event Engine Pipeline 框架

**目标**：构建 ingest → process → store 的事件处理管线

**涉及文件**：
- `tools/eaasp-l4-orchestration/src/eaasp_l4_orchestration/event_engine.py` — 新建
- `tools/eaasp-l4-orchestration/src/eaasp_l4_orchestration/event_handlers.py` — 新建

**实现要点**：
1. `EventEngine` 类：
   - `ingest(event: Event) → ProcessedEvent` — 接收+标准化
   - `process(events: list[Event]) → list[ProcessedEvent]` — 批处理管线
   - `start() / stop()` — 生命周期管理
2. Handler 接口（对齐 ADR-V2-003）：
   ```python
   class EventHandler(Protocol):
       async def handle(self, event: Event) -> Event | None: ...
   
   class Ingestor(EventHandler): ...
   class Deduplicator(EventHandler): ...
   class Clusterer(EventHandler): ...
   class Indexer(EventHandler): ...
   ```
3. Pipeline 按顺序执行 handler chain：Ingest → Dedup → Cluster → Index
4. 异步处理：收到事件后放入队列，后台 worker 消费

**测试**：6-8 个测试（pipeline execution / handler chain / async queue / error handling）

**前置**：S1 完成

### S3.T2 — Deduplicator + Clusterer 默认实现

**目标**：提供 Phase 1 可用的去重和聚类实现

**涉及文件**：
- `tools/eaasp-l4-orchestration/src/eaasp_l4_orchestration/event_dedup.py` — 新建
- `tools/eaasp-l4-orchestration/src/eaasp_l4_orchestration/event_cluster.py` — 新建

**实现要点**：
1. `TimeWindowDeduplicator`：
   - 相同 (session_id, event_type, tool_name) 在 N 秒窗口内合并

   - **不做** topology-aware 聚类（需要 ontology，Phase 5）
