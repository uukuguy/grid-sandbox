## 3. CLI-04 (TUI key_handler.rs 拆分) 讨论 — D.1+D.2 / D.3 / D.4

### D.1+D.2 — 模块切分维度 + Dispatcher 形式 (捆绑决策)

| Option | Pros | Cons | the agent's Recommendation |
|--------|------|------|------------------------|
| (a) 按模式切 + match dispatcher | ratatui 生态 + vim/helix/zellij TUI 都按模式切, mental model 对齐, 单文件 dispatcher "哪个 mode 处理哪些键" 一眼看完 | 跨模式共享逻辑要抽 `common.rs` | ✓ Recommended |
| (b) 按动作切 + match dispatcher | 相关动作集中 | 跨 mode 时变得拧巴, 每个动作文件都判 mode | |
| (c) 按模式切 + trait `KeyHandler` dyn dispatch | OOP 风格 | ≤10 模式场景下是 over-engineering | |

**User's choice:** 同意你的推荐 → (a) 按模式切 + match dispatcher

### D.3 — Invariant 表 schema 详细程度?

| Option | Pros | Cons | the agent's Recommendation |
|--------|------|------|------------------------|
| (a) 完整版 (Mode/Key/Action/Pre-condition/Post-condition 5 列) | 防 silent regression 最严密 | 200 行变成 400 行 | |
| (b) 轻量版 (Mode/Key → Action 2 列) | 你不是写正式 spec, 是给自己留 source-of-truth, 测试当回归保险 | 没有 pre/post-condition | ✓ Recommended |

**User's choice:** 同意你的推荐 → 轻量版

### D.4 — Integration test harness 选型?

| Option | Pros | Cons | the agent's Recommendation |
|--------|------|------|------------------------|
| (a) 模拟 KeyEvent 直接灌 `dispatch(mode, key)` 断言 Action enum | 轻量, 直接验 dispatcher 行为 | 不验 render | ✓ Recommended (作为 integration 主体) |
| (b) ratatui `TestBackend` 跑全 TUI assert render | 完整 e2e | 重, 需 mock terminal buffer, ratatui 没官方 testing harness | |
| (c) 只单元测, 不 integration | 最轻 | 不抓跨模式序列 bug | ✓ Recommended (作为 unit 主体, 配合 (a) integration ≥2 个) |

**User's choice:** 同意你的推荐 → (a) + (c) 组合, 单元每 mode ≥3 个 + integration ≥2 个跨模式序列, 不用 (b)
