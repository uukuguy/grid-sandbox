## 1. Executive Summary

Five things the planner needs to know in 30 seconds:

1. **CLI-01 is mostly help-output cosmetics + a CI gate.** The `cli-` prefix exists ONLY as Makefile target names (`make cli-ask`, `make cli-doctor`); these targets already invoke the correct subcommands (`grid ask`, `grid doctor`). There is **zero** `cli-ask`/`cli-session`/`cli-config`/`cli-doctor` literal in scripts/, examples/, tests/, lang/, tools/. The deprecation-alias work (D-03/D-04) is therefore a **pre-emptive guard** — implement aliases via `clap::Subcommand` `visible_aliases`, and the in-tree consumer scan (D-06) will report 0 from day one. The Makefile targets themselves are the only consumer; rename them in T1 of the plan.

2. **CLI-02 ChunkType render lives in `ui/streaming.rs`, NOT `output/text.rs`.** The orchestrator's note about `output/text.rs` having ChunkType arms is incorrect: `output/text.rs` is 28 LOC of generic helpers and `output/stream_json.rs` is 10 LOC. The actual streaming render consumes Rust-side `AgentEvent` (NOT proto-wire `ChunkType`) from grid-engine. Proto `ChunkType` only flows in the gRPC client path (server-side, `tools/eaasp-cli-v2/.../cmd_session.py`). Grid-cli renders `AgentEvent::TextDelta/ToolStart/ToolResult/Error/Done/WorkflowContinuation/IterationStart/IterationEnd/...` — D-07..D-12 implementations live in `ui/streaming.rs::render_text_stream` (currently 122 LOC, will grow to ~300).

3. **CLI-04 Mode taxonomy is NOT vim-style 6 modes.** The actual TUI state machine (in `tui/app_state.rs`) is REPL-conversational, not modal-editor. There is no `pub enum Mode { Normal, Insert, Command, Visual, Help, Dialog }`. There is `OverlayMode { None, AgentDebug, Eval, SessionPicker }` + cross-cutting sub-states (`vim.enabled` Vim sub-mode, `history_search.active`, `model_selector.visible`, `pending_approval.is_some()`). The 1556 LOC `key_handler.rs` is structured as a chain of early-return guards that route to inline handlers per active sub-state. **D-16/D-18 file structure (`normal.rs/insert.rs/command.rs/visual.rs/help.rs/dialog.rs`) needs reinterpretation** — see §6 Critical Finding for two viable mappings the planner must choose between (recommendation: bounce to user, or pick the "early-return state" mapping that follows the existing code shape).

4. **CLI-05 needs SQLite migration v9.** Existing schema (`migrations.rs::migration_v1`) has `sessions(session_id, user_id, sandbox_id, created_at)` only — no `last_turn_at`, no `deleted_at`. Migration framework (Migration struct + `migration_v1..v8` already exist) is the standard pattern; add `migration_v9` with `ALTER TABLE sessions ADD COLUMN last_turn_at INTEGER; ALTER TABLE sessions ADD COLUMN deleted_at INTEGER NULL;` + populate `last_turn_at` from MAX(session_messages.created_at) per session. Backwards-compatible (NULL deleted_at = active row).

5. **CLI-06 has a clean extension shape.** `commands/doctor.rs` already has `CheckResult { name, status, message, fix_hint, repair_result }` + `CheckStatus { Pass, Warn, Fail }`. Add `severity: Severity { Error, Warn, Info }` field for D-27 (note: existing `Status` and new `Severity` are different concepts — Status is the result, Severity is the impact). Add `--json` and `--offline` flags + 6 new check functions (one each for the 5 ROADMAP must-checks + proto-cli-sync). The `proto-cli-sync` check is the most novel: read the compile-time `grid_runtime::proto::ChunkType` enum (via `strum::IntoEnumIterator` macro on the proto-generated type — alternative: parse `grid_runtime::proto::ChunkType::is_valid()` value space 0..=7) and compare against the AgentEvent → ChunkType mapping table that grid-runtime's `service.rs::chunk_type_to_proto` uses.

**Primary recommendation:** Single plan with 6-8 tasks, NOT two plans. Tasks T1-T2 mechanical (rename + studio confirm), T3-T7 reasoning-heavy (UX + dispatcher + migration + doctor). Plan-checker may suggest a split if it finds T6 (key_handler split) too risky to bundle.

### clap 4.5 derive grouping pattern (verified via context7 `/clap-rs/clap`)

The `#[command(next_help_heading = "...")]` attribute can decorate Subcommand enum variants in clap 4.5 — it sets the section title under which subsequent variants render in `--help`. Behavior caveat from [clap-rs/clap#5828](https://github.com/clap-rs/clap/issues/5828): the heading attribute on the **first variant** of a group sets the group; subsequent variants inherit until the next `next_help_heading`. There is a known UX quirk (heading shows once at the boundary), but for our 5-group case this is exactly what we want.

```rust
// Pattern for crates/grid-cli/src/lib.rs Cli::Commands
#[derive(Parser)]
pub enum Commands {
    // ── INTERACT ────────────────────────────────────
    /// Start interactive REPL session
    #[command(next_help_heading = "INTERACT")]
    Run { /* ... */ },
    /// Send a single query (headless mode)
    Ask { /* ... */ },
    /// Manage agents (run agent loops, list agents)
    Agent { #[command(subcommand)] action: AgentCommands },

    // ── SESSION ─────────────────────────────────────
    /// Manage sessions (list / resume / kill)
    #[command(next_help_heading = "SESSION")]
    Session { #[command(subcommand)] action: SessionCommands },

    // ── CONFIG ──────────────────────────────────────
    /// Configuration management
    #[command(next_help_heading = "CONFIG")]
    Config { #[command(subcommand)] action: ConfigCommands },
    /// Initialize Grid project in current directory
    Init,
    /// Generate shell completions
    Completions { #[command(subcommand)] action: CompletionsCommands },

    // ── DIAGNOSTIC ──────────────────────────────────
    /// Run health diagnostics
    #[command(next_help_heading = "DIAGNOSTIC")]
    Doctor {
        /// Output as JSON for CI consumption
        #[arg(long)]
        json: bool,
        /// Skip network checks (offline mode)
        #[arg(long)]
        offline: bool,
        // (D-29: --repair / --fix removed)
    },

    // ── RESOURCES ───────────────────────────────────
    /// Manage MCP servers
    #[command(next_help_heading = "RESOURCES")]
    Mcp { #[command(subcommand)] action: McpCommands },
    /// Manage memory
    Memory { #[command(subcommand)] action: MemoryCommands },
    /// Manage tools
    #[command(name = "tool")]
    Tool { #[command(subcommand)] action: ToolsCommands },
    /// Manage skills
    Skill { #[command(subcommand)] action: SkillCommands },
    /// Show/manage GridRoot paths
    Root { #[command(subcommand)] action: RootCommands },
    /// Sandbox execution environment diagnostics
    Sandbox { #[command(subcommand)] action: SandboxCommands },
    /// Evaluation management
    Eval { #[command(subcommand)] action: EvalCommands },

    // ── AUTH ────────────────────────────────────────
    /// Manage API credentials (login/status/logout)
    #[command(next_help_heading = "AUTH")]
    Auth { #[command(subcommand)] action: AuthCommands },
}
```

**Note:** `Doctor` field shape changes from `{ repair: bool }` (current) to `{ json: bool, offline: bool }` (per D-26/D-28/D-29) — this is a breaking change to the public CLI surface, but `repair` is gone per D-29 anyway, and current `--repair` was a single-flag oversight.

### Deprecation alias pattern for `cli-*` prefix (D-03)

clap 4.5 supports `visible_aliases` (shows in help) and `aliases` (hidden, accepted but undocumented). For deprecation, use `aliases` (hidden) so help is clean, and add a stderr warning at parse time. clap doesn't natively support "matched alias name detection" — we need a pre-parse arg sniff:

```rust
// In main.rs, before Cli::parse():
fn warn_deprecated_aliases(args: &[String]) {
    const DEPRECATED: &[(&str, &str)] = &[
        ("cli-ask", "ask"),
        ("cli-session", "session"),
        ("cli-config", "config"),
        ("cli-doctor", "doctor"),
        // add new aliases here as needed
    ];
    if let Some(arg1) = args.get(1) {
        for (old, new) in DEPRECATED {
            if arg1 == old {
                eprintln!(
                    "warning: '{}' is deprecated, use '{}' instead \
                     (will be removed when no in-tree consumers remain)",
                    old, new
                );
            }
        }
    }
}

#[tokio::main]
async fn main() -> ExitCode {
    let args: Vec<String> = std::env::args().collect();
    warn_deprecated_aliases(&args);
    // rewrite args[1] to canonical name, then parse
    // ...
}
```

Then add `#[command(aliases = &["cli-ask"])]` to each renamed variant. Hidden aliases keep `--help` clean while still accepting the old name. Approach is parallel to git's deprecated alias handling (e.g., `git fetch --no-tags` warning patterns).

## 4. CLI-02 — ChunkType streaming render

### Critical correction to orchestrator's framing

**Grid-cli does NOT render proto ChunkType.** The render layer consumes `grid_engine::agent::AgentEvent` (Rust-side enum), which is a superset of proto `ChunkType` (8 wire values) plus engine-internal events (TokenBudgetUpdate / Typing / IterationStart / IterationEnd / ApprovalRequired / SecurityBlocked / ToolProgress / Completed / PlanUpdate / EmergencyStopped / RetryingMalformedToolCall / WorkflowContinuation / SubAgentEvent / CostUpdate / InteractionRequested / Autonomous* / ContextCompacted / ContextDegraded / MemoryFlushed).

ChunkType-equivalent AgentEvent variants relevant to D-08..D-10:
- `TextDelta { text }` ↔ `CHUNK_TYPE_TEXT_DELTA` — typewriter target
- `ThinkingDelta { text }` / `ThinkingComplete` ↔ `CHUNK_TYPE_THINKING` — already dim, skip typewriter
- `ToolStart { tool_id, tool_name, input }` ↔ `CHUNK_TYPE_TOOL_START` — collapse target (D-08)
- `ToolResult { tool_id, tool_name, output, success }` ↔ `CHUNK_TYPE_TOOL_RESULT` — collapse target (D-08)
- `Done` ↔ `CHUNK_TYPE_DONE` — newline + flush
- `Error { message }` ↔ `CHUNK_TYPE_ERROR` — red highlight (D-10)
- `WorkflowContinuation { attempt, max_attempts }` ↔ `CHUNK_TYPE_WORKFLOW_CONTINUATION` — dim grey marker (D-09)

The proto wire mapping happens at gRPC boundary in `crates/grid-runtime/src/service.rs::chunk_type_to_proto` (already implemented per ADR-V2-021). Grid-cli's `ui/streaming.rs` is a parallel consumer of the same `AgentEvent` upstream.

### crossterm + owo-colors patterns (already wired in codebase)

`grid-cli` uses `owo-colors = "4"` (in deps), which is the modern fork of `colored`. Examples already in `ui/streaming.rs`:
```rust
use owo_colors::OwoColorize;
eprintln!("  {} {}", "✔".green(), preview.dimmed());
eprintln!("{}", format!("Error: {}", message).red().bold());
```

For dim grey (D-09), `.dimmed()` is the canonical owo-colors method (renders as ANSI `\x1b[2m`). For red (D-10), `.red().bold()` is already used.

### `NO_COLOR` + `--no-color` (D-11) — already wired

`crates/grid-cli/src/main.rs:74`:
```rust
color: !cli.no_color && std::io::IsTerminal::is_terminal(&std::io::stdout()),
```

This already honors `--no-color` and TTY detection. Missing: respect `NO_COLOR` env var per [no-color.org](https://no-color.org/). Fix:
```rust
color: !cli.no_color
    && std::env::var("NO_COLOR").map_or(true, |v| v.is_empty())
    && std::io::IsTerminal::is_terminal(&std::io::stdout()),
```

`std::io::IsTerminal` is stable since Rust 1.70; no `is-terminal`/`atty` crate needed (they're deprecated in favor of std).

```rust
// In Cli struct (lib.rs):
/// Force typewriter streaming output (default: off; auto-on if TTY)
```

```rust
// crates/grid-cli/src/error.rs
use thiserror::Error;

/// CLI-layer error types that map to specific sysexits exit codes.
/// Inner errors (anyhow chain) are preserved for cause display via --verbose.
#[derive(Debug, Error)]
pub enum GridCliError {
    #[error("usage error: {0}")]
    Usage(String),

    #[error("config not found: {path}")]
    ConfigMissing { path: String },

    #[error("config invalid: {0}")]
    ConfigInvalid(String),

    #[error("authentication failed: {0}")]
    AuthFailed(String),

    #[error("provider/network error: {0}")]
    Unavailable(String),

    #[error("internal error: {0}")]
    Internal(String),
}

impl GridCliError {
    pub fn exit_code(&self) -> i32 {
        use crate::exit_code::*;
        match self {
            Self::Usage(_) => USAGE,
            Self::ConfigMissing { .. } => NOINPUT,
            Self::ConfigInvalid(_) => CONFIG,
            Self::AuthFailed(_) => NOPERM,
            Self::Unavailable(_) => UNAVAILABLE,
            Self::Internal(_) => SOFTWARE,
        }
    }
}

/// Classify any anyhow::Error chain into a sysexits exit code.
/// Walks `error.chain()` looking for typed `GridCliError` first;
/// falls back to string-pattern heuristics for `reqwest::Error`, `tonic::Status`, etc.
pub fn classify(err: &anyhow::Error) -> i32 {
    // First: typed downcast
    for cause in err.chain() {
        if let Some(gce) = cause.downcast_ref::<GridCliError>() {
            return gce.exit_code();
        }
    }
    // Fallback heuristics for upstream untyped errors:
    let msg = err.to_string().to_lowercase();
    if msg.contains("api key") || msg.contains("unauthor") || msg.contains("authentication") {
        return crate::exit_code::NOPERM;
    }
    if msg.contains("timeout") || msg.contains("connection refused") || msg.contains("unavailable") {
        return crate::exit_code::UNAVAILABLE;
    }
    if msg.contains("not found") && msg.contains(".yaml") {
        return crate::exit_code::NOINPUT;
    }
    crate::exit_code::SOFTWARE
}
```

## 10. Common Pitfalls

### Pitfall 1: clap `next_help_heading` with subcommand groups
**What goes wrong:** Heading attribute doesn't render correctly when split across many subcommand variants — sometimes only the first heading shows, sometimes none.
**Why it happens:** Per [clap-rs/clap#5828](https://github.com/clap-rs/clap/issues/5828), clap's heading propagation across enum variants has UX quirks. The attribute MUST be on the first variant of each group; subsequent variants in the same group should NOT repeat it.
**How to avoid:** Test the actual help output with a snapshot test (`grid --help` → string assertion contains "INTERACT" / "SESSION" / etc. headers in order). Bake this into T2.
**Warning signs:** `cargo run -p grid-cli -- --help` shows ungrouped subcommand list.

### Pitfall 2: `IsTerminal` shadowing prior crate `is-terminal` / `atty`
**What goes wrong:** Adding `is-terminal` or `atty` crate when stdlib `std::io::IsTerminal` (stable 1.70+) is already used in `main.rs:74`.
**Why it happens:** Old tutorial blog posts still recommend `atty` (now archived).
**How to avoid:** Stick with `std::io::IsTerminal::is_terminal(&std::io::stdout())`. No new crate needed.
**Warning signs:** New `atty` or `is-terminal` line in Cargo.toml during code review.

### Pitfall 3: `OutputFormat::Text` being chosen when stdout is piped
**What goes wrong:** User does `grid ask "hello" | tee out.txt` and gets ANSI escape codes in the file.
**Why it happens:** Code uses `--output text` default (text format) regardless of TTY state; only `color` is auto-disabled. ANSI escapes outside color (e.g., `.bold()`, `.dimmed()`) still render.
**How to avoid:** When `is_terminal == false`, force `config.color = false` (already done) AND consider plain-text mode for ALL ANSI codes — owo-colors respects `color=false` for `.green()` etc. but not all helper functions. Audit `ui/streaming.rs` calls.
**Warning signs:** Piped output contains `\x1b[`.

### Pitfall 4: D-19 invariant table going stale during the split
**What goes wrong:** T0 commits the invariant table; T3-T5 split files; one mode handler subtly changes behavior (e.g., a key in `OverlayMode::Eval` was meant for `Up` but new file accidentally uses `Down`).
**Why it happens:** No mechanical link between table and code — the table is human-maintained.
**How to avoid:**
- Make the table a `mod tests` doc-comment block in `mod.rs` so it lives WITH the dispatcher
- Add one integration test per row: `dispatch(state, key) → action` — at least for the 20 highest-traffic rows
- T6 verification step: re-grep the source post-split, regenerate the table, diff against the T0-committed version
**Warning signs:** Test fails after split + reviewer says "doesn't this look like it does X?"

### Pitfall 5: SQLite migration_v9 ALTER on a busy DB
**What goes wrong:** `ALTER TABLE sessions ADD COLUMN` is fast but the `UPDATE sessions SET last_turn_at = (SELECT MAX...)` backfill query scans all session_messages — on a long-running grid.db with thousands of messages, this can take seconds.
**Why it happens:** SQLite ALTER is atomic; UPDATE with subquery isn't.
**How to avoid:** Either (a) accept the one-time delay — SQLite is fast on modern hardware, (b) split into two migrations (v9 schema add only, v10 backfill), or (c) skip backfill: leave NULL last_turn_at and let runtime populate on next session activity.
**Recommendation:** Option (a) — keep migration_v9 atomic. Add INDEX BEFORE the UPDATE so backfill uses it.
**Warning signs:** First `grid` invocation after upgrade hangs >5s.

### Pitfall 6: gRPC ping with default tonic timeout vs explicit 5s
**What goes wrong:** `RuntimeServiceClient::connect()` without explicit timeout hangs indefinitely on a half-open TCP connection.
**Why it happens:** tonic 0.x default timeout is "no timeout" — defers to OS keepalive (60s+).
**How to avoid:** Always wrap with `tokio::time::timeout(Duration::from_secs(5), ...)` AND set `Endpoint::timeout(Duration::from_secs(5))`. Both layers (tokio-level and tonic-level) are needed because tonic timeout applies to the request, not the connect.
**Warning signs:** `grid doctor` hangs when L1 runtime is half-up.

### Pitfall 7: Phase 5.0 / 5.1 lessons (relevant to 5.2)
- **Phase 5.0 GA1 dead code lesson:** `D134 GA1` was dead code in scope guards — verify by grep + manual read, don't trust tests alone. **Apply to 5.2:** when D-19 invariant table is generated, manually walk every entry; don't rely on tests.
- **Phase 5.1 cross-AI review lesson:** four reviewers caught B1 pre-committed verdict in T4 hardcoded "两腿都推进" Path 1 fix. **Apply to 5.2:** if T6 (key_handler split) is high-risk, mark `review_protocol: superpowers-two-stage` in PLAN.md frontmatter (per Phase 4 REVIEW_POLICY.md precedent).

## 12. Open Questions

### O1: TUI key_handler mode taxonomy mismatch (CLI-04 D-18)

**The question:** D-18 locks file structure as `normal/insert/command/visual/help/dialog.rs` but the actual TUI is REPL-conversational, not vim-modal — there is no `Mode::Normal` enum. §6 proposes Option A (rename files to match real sub-states: `normal/insert/command/overlay/approval/vim/search/selector/common.rs`) vs Option B (literal D-18 names with mismatched contents).

**What we know:** code reading confirms `OverlayMode { None, AgentDebug, Eval, SessionPicker }` + cross-cutting sub-states (vim, history_search, model_selector, pending_approval). 1556 LOC `key_handler.rs:221-410` shows early-return chain dispatch.

**What's unclear:** discuss-phase locked D-18 file names assuming a vim-style mode hierarchy that doesn't exist. Both Option A (faithful to code) and Option B (faithful to D-18 letter) are implementable.

**Recommendation:** Bounce to user briefly during plan-phase: "D-18 file names assume vim modes; codebase has overlay+sub-states. Use real names (Option A) or keep D-18 literal (Option B)?" If user is unavailable, planner picks Option A — file names should reflect contents.

### O2: gRPC reachability tier-detection (CLI-06 D-24/D-25 + ADR-V2-025)

**The question:** Should `grid doctor` ping ALL 7 runtimes or just the configured one? ADR-V2-025 specifies tier-aware severity, but doesn't mandate which runtime grid-cli targets.

**What we know:** Default grid-runtime addr = `127.0.0.1:50051`. Other runtimes have different ports (50052/53/54). User's `config.yaml` may specify which one.

**What's unclear:** is "L1 runtime gRPC reachability" check a single-target ping (configured runtime), a multi-target sweep (all 7), or user-specified `--runtime <name>`?

**Recommendation:** Default = ping configured/default runtime (single target). Add `--all-runtimes` flag for full sweep. Severity per ADR-V2-025 tier: configured runtime fail = Error if Primary tier, Warn if Sample/Reference, Info if Frozen.

### O3: typewriter delay configurability (Claude's discretion per CONTEXT.md)

**The question:** Should typewriter delay (default 30ms/char) be exposed as `--typewriter-delay <ms>` flag?

**What we know:** Industry default 25-40ms; Grid's primary use case includes machine consumption (test harness piping stdout) where typewriter is off anyway.

**What's unclear:** is 30ms hardcoded sufficient, or do power users want tuning?

**Recommendation:** Hardcode 30ms for v3.1; revisit if user feedback requests tuning. Keeps surface small.
