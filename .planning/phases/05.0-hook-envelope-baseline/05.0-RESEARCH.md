# Phase 5.0: Hook Envelope Baseline - Research

**Researched:** 2026-05-19
**Domain:** Hook envelope contract (ADR-V2-006), Stop envelope parity, shipped hook script maintenance
**Confidence:** HIGH (verified against source code in all four domains)

## Summary

Phase 5.0 is a baseline health patch addressing three categories of drift between the ADR-V2-006 contract and shipped artifacts. GA1 (`.output.*` dead code) is real and confirmed: both shipped hook scripts check `.output.evidence_anchor_id` and `.output.draft_memory_id` but the Rust engine (`HookContext::to_stop_envelope`, lines 416-428) and Python runtime (`_dispatch_scoped_stop`, lines 1029-1036) emit flat top-level fields only. The dual-path checks are dead code under the current contract. GA2 is a false alarm: `reason`, `stop_type`, and `full_completion` fields do not exist in the ADR-V2-006 canonical spec (any version), the Rust `HookContext` struct, or the Python `_dispatch_scoped_stop` function. The CONTEXT.md reference to them appears to be a documentation error. GA3 (D134 audit) is straightforward: hooks are shipped and the test suite has Stop coverage, but the D134 claim of completeness needs an explicit written verification pass.

**Primary recommendation:** Implement GA1 as Option 1 (full deletion of dead dual-path checks with explanatory comments). Treat GA2 as closed -- no action needed since those fields are not in the ADR. For GA3, do the D134 audit as a verification task, not an implementation task.

## GA1: `.output.*` Dual-Path Dead Code

### Finding: Dead code confirmed in both shipped hooks

**Location 1:** `examples/skills/skill-extraction/hooks/check_final_output.sh` [VERIFIED: source code read]

The `has_output_both()` function reads:
```bash
has_output_both() {
  echo "$input" | jq -e '(.output // {} | .draft_memory_id // "") | length > 0'
  echo "$input" | jq -e '(.output // {} | .evidence_anchor_id // "") | length > 0'
}
```
These check `.output.draft_memory_id` and `.output.evidence_anchor_id`. Both paths are dead under the ADR-V2-006 flat schema.

The `has_top_both()` function checks the correct flat paths:
```bash
has_top_both() {
  echo "$input" | jq -e '(.draft_memory_id // "") | length > 0'
  echo "$input" | jq -e '(.evidence_anchor_id // "") | length > 0'
}
```
The `if has_top_both || has_output_both` guard (line ~45) means the `.output.*` checks never contribute to the decision -- the flat path is always checked first with `||`, and `.output.*` only runs if flat fails.

**Location 2:** `examples/skills/threshold-calibration/hooks/check_output_anchor.sh` [VERIFIED: source code read]

The `has_output_anchor()` function reads:
```bash
has_output_anchor() {
  echo "$input" | jq -e '(.output // {} | .evidence_anchor_id // "") | length > 0'
}
```
Dead under ADR-V2-006.

Both hooks carry a comment explaining the dual-path as "forward/backward compat" for "older engines" that nested output under `.output.*`. This compat layer is no longer needed.

### Evidence: Engine never emits `.output.*`

**Rust `HookContext::to_stop_envelope()`** (`crates/grid-engine/src/hooks/context.rs`, lines 416-428) [VERIFIED: source code read]:

```rust
fn to_stop_envelope(&self) -> serde_json::Value {
    serde_json::json!({
        "event": "Stop",
        "session_id": self.session_id.clone().unwrap_or_default(),
        "skill_id": self.skill_id.clone().unwrap_or_default(),
        "draft_memory_id": self.draft_memory_id.clone().unwrap_or_default(),
        "evidence_anchor_id": self.evidence_anchor_id.clone().unwrap_or_default(),
        "created_at": self.created_at.clone().unwrap_or_else(current_iso8601_z),
    })
}
```

Six flat top-level fields only. No `.output` wrapper. `draft_memory_id` and `evidence_anchor_id` are top-level keys, not nested.

**Python `_dispatch_scoped_stop()`** (`lang/claude-code-runtime-python/src/claude_code_runtime/service.py`, lines 1029-1036) [VERIFIED: source code read]:

```python
envelope = {
    "event": "Stop",
    "session_id": sid,
    "skill_id": skill_id,
    "draft_memory_id": "",   # not populated from actual memory
    "evidence_anchor_id": "",  # not populated from actual anchor
    "created_at": datetime.now(timezone.utc).strftime("%Y-%m-%dT%H:%M:%SZ"),
}
```

Identical flat shape. No `.output` nesting. Both `draft_memory_id` and `evidence_anchor_id` are always `""` in the Python runtime -- the actual values are not threaded through from session state.

### Options for Planner

| Option | Approach | Risk | Recommendation |
|--------|----------|------|----------------|
| 1 (recommended) | Full deletion of `.output.*` checks with explanatory comment | Low -- no runtime emits `.output.*` | Delete, add comment citing ADR-V2-006 and Phase 5.0 date |

| 3 | Comment-out with `# DEPRECATED: removed in Phase 5.0 per ADR-V2-006` | Low | Acceptable alternative to full deletion |

**Planner note:** Option 1 is cleanest. If any external integration (nanobot-runtime, future goose-runtime) ever emitted `.output.*`, this would surface as a hook failure and could be re-added. The current state is a maintenance burden with zero benefit.

## GA2: Stop Envelope Fields -- `reason`/`stop_type`/`full_completion`

### Finding: These fields are NOT in ADR-V2-006 or either runtime

**Critical clarification:** The CONTEXT.md references ADR-V2-006 §2.3 as including `reason`, `stop_type`, and `full_completion` fields. **This is a documentation error.** Reading the ADR verbatim:

**ADR-V2-006 §2.3 canonical Stop envelope example:**
```json
{
  "event": "Stop",
  "session_id": "sess-abc123",
  "skill_id": "threshold-calibration",
  "draft_memory_id": "mem-42",
  "evidence_anchor_id": "anchor-99",
  "created_at": "2026-04-15T10:31:00Z"
}
```

**ADR-V2-006 §2.4 schema rules for Stop:**
> `draft_memory_id` and `evidence_anchor_id` are OPTIONAL. When absent, the runtime MUST emit empty string `""` (not `null`, not missing).

Six total fields for Stop envelope: `event`, `session_id`, `skill_id`, `draft_memory_id`, `evidence_anchor_id`, `created_at`. **No `reason`, `stop_type`, or `full_completion`.**

**Rust `HookContext` struct** (`crates/grid-engine/src/hooks/context.rs`, lines 44-62) [VERIFIED: source code read]:
- `draft_memory_id: Option<String>` -- present
- `evidence_anchor_id: Option<String>` -- present
- No `reason` field
- No `stop_type` field
- No `full_completion` field

**Python `_dispatch_scoped_stop`** (`service.py` lines 1029-1036) [VERIFIED: source code read]:
- Same six fields, no additions

**`hook_envelope_parity_test.rs` coverage** [VERIFIED: source code read]:
- `hook_envelope_stop_matches_adr_example()` (line 111) -- tests the six ADR fields
- `hook_envelope_stop_absent_optionals_serialize_as_empty_string()` (line 137) -- tests that absent optionals serialize as `""`, not `null` or absent

The parity tests are **complete for the current ADR**. There is no gap.

### Implication for Planning

The "missing coverage" described in GA2 does not exist under the current ADR. Two paths forward:

| Path | Description | When to use |
|------|-------------|-------------|
| A (recommended) | Treat GA2 as closed -- no action. The ADR does not define these fields. | Default assumption unless user corrects |
| B | Investigate whether `reason`/`stop_type`/`full_completion` should be added to the ADR (Phase 3+ work) | Only if user confirms these fields are intended |

If Path B is chosen: the work becomes an ADR amendment, not a test-addition task. The planner should note this as out-of-scope for Phase 5.0 baseline maintenance.

## Common Pitfalls

### Pitfall 1: Treating GA2 as a real gap
**What goes wrong:** Planner spends effort adding tests for `reason`/`stop_type`/`full_completion` fields that do not exist in the ADR.
**Why it happens:** CONTEXT.md references these fields as if they are in ADR-V2-006 §2.3, but they are not.
**How to avoid:** Verify field existence in source code (Rust `HookContext` struct, Python `_dispatch_scoped_stop` envelope dict) before writing test tasks. The ADR canonical examples are authoritative.

### Pitfall 2: Deleting `.output.*` without explanatory comments
**What goes wrong:** Future developers see bare deletion and re-add the compat paths as "forward compat."
**Why it happens:** The shipped hooks had comments explaining the dual-path as "backward compat for older engines." Deleting without replacement comment loses this context.
**How to avoid:** Add a replacement comment in each hook explaining that `.output.*` was removed in Phase 5.0 per ADR-V2-006 flat-top-level schema.

### Pitfall 3: Confusing feature gaps with contract gaps
**What goes wrong:** Treating Python runtime always-emitting `""` for `draft_memory_id`/`evidence_anchor_id` as a D120 contract violation.
**Why it happens:** The contract says fields MUST be present (satisfied) and MUST be `""` when absent (satisfied -- the runtime treats all values as absent). The feature of actually populating memory IDs is out of contract scope.
**How to avoid:** Separate contract compliance verification from feature completeness verification. D120 = contract compliance. Memory/anchor population = separate feature.

### Pitfall 4: D134 as implementation task vs verification task
**What goes wrong:** Treating D134 audit as requiring code changes.
**Why it happens:** D134 is a "已落盘" (landed) item -- hooks are already shipped.
**How to avoid:** D134 audit is a read-and-confirm task, not an implementation task. Run existing tests and write a closure statement.
