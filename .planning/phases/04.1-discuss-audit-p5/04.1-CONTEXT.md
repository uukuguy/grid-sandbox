### §P5 解读基线 (gray area 2 — Option C, 平行交叉)

- **D-B-01**: Audit doc 同时审 **ADR-V2-023 §P5 4 条 trigger** AND **`.planning/phases/4.1-PRE-AUDIT-NOTES.md` §F Q1-Q4 audit agenda**
- **D-B-02**: 平行交叉对照 —— audit doc 在每条 §P5 trigger 后显式写"**Cross-ref**: 此 trigger 在 §F 框架下对应 Qx 的哪个面"; 在每条 §F Qx 后显式写"**Cross-ref**: 此问题在 §P5 框架下对应 trigger N 的 vintage 表述"
- **D-B-03**: §P5 trigger 1 (EAASP 项目延迟/停滞/定价不合理) → §C.2 baseline 已说明此条在同仓孵化下"语义弱化",audit 必须显式说"**作为外部风险评估已弱化, 但作为自评内部节奏指标可保留**" — 不简单标 N/A
- **D-B-04**: §F 是 baseline §F (commit `e1f27df`) 写入的 4 个真问题, audit doc 把它们当作"内部评估浮出的具体决策门"对待, 不只是 Claude 的素材 — user 在 baseline 落盘已视为必须解
- **D-B-05**: Audit 不**否决** §P5 4 条原文(尊重 ADR-V2-023 字面), 但允许"**部分弱化**"措辞 + 提议 ADR-V2-024 重新框定

### 推荐结论形态 (gray area 3 — Option B, 三选一 + 框架修订建议)

- **D-C-01**: Audit doc 必须给出 ROADMAP SC#2 要求的"**走 Leg A 硬化 / 走 Leg B 激活 / 两条腿都推进**"三选一推荐, 措辞明确不含糊
- **D-C-02**: 同时**额外**给"**框架修订建议**"段, 提议 Phase 4.2 ADR-V2-024 把 Leg A/B 二元切换为"engine vs data/integration"+"Grid 全栈 + EAASP 引擎层 vs 数据/集成横切"双轴模型(per §C.1 + §F Q1)
- **D-C-03**: 框架修订建议**不**是 audit 的主推论, 是附加观察。三选一 + 框架修订建议**正交**: 即便 ADR-V2-023 字面框架保留, 三选一推荐仍然有效
- **D-C-04**: SC#2 严格满足三选一即可, 框架修订建议是 audit 真价值的体现, 不是越权 — Phase 4.2 决定是否采纳框架修订

### GOVERNANCE-02/03 plumbing 验证 (gray area 4 — Option A, 顺势观察)

- **D-D-01**: Phase 4.1 不专门拆"GOVERNANCE-02 plumbing dry-run" task。plumbing 在 audit 任务自然走 plan-phase + execute-phase + verify 链路时被动验证
- **D-D-02**: Phase 4.0 已实证 plumbing 端到端 OK (commit `c12f425`), 4.1 是第二次验证 — 重点观察"**audit 这种 design-heavy 性质 task 跑 GSD 是否仍顺畅**", 不是再次端到端冒烟
- **D-D-03**: SC#3 "记录任何不顺手处到 `docs/dev/WORK_LOG.md` 顶部 'GSD adoption notes' 块" — 在 phase 末 verify 时由 verifier agent 读 phase 全过程, 自动产出 adoption notes 候选清单, 由 user 末端审阅入 WORK_LOG (不在 phase 中段实时记录, 避免分心)
- **D-D-04**: SC#4 `/gsd-resume-work` 触发时机 = **Phase 4.1 audit 写到一半时显式 `/clear`** (人为触发, 非"等自然中断"), 然后 `/gsd-resume-work`, 检查是否完整恢复 audit 上下文。**这是真实测试, 不是 dry-run** — Phase 4.1 long-running audit 是合适场景

### Audit Evidence 标准 (gray area 5 — Option B, 多源)

- **D-E-01**: 每条 audit 结论的 evidence 来源 = **三档之一**:
  - **代码/文件级 evidence** — git log / file existence / 具体 LOC 数 / commit hash / ADR id
  - **User 直接陈述** — Phase 4.1 socratic 时锁定 (audit 内 inline quote, 标 "user 2026-04-27 socratic"), 或 4.0 / pre-4.0 baseline 已锁 (引用 4.1-PRE-AUDIT-NOTES.md §X.Y)
  - **ADR 文本 cross-ref** — 引用 ADR-V2-023 / V2-021 / V2-006 等具体 §X.Y, 必须 file:line 引用避免 paraphrase 漂移
- **D-E-02**: Audit 不试图自己推断业务事实 (eg. 是否有 ≥2 客户要求 Grid 独立 — 只有 user 知道)。这种条目走"**user 陈述待补**"标记,phase 4.1 socratic 时由 user 给真实数字
- **D-E-03**: 每条 audit point 的 verdict (yes/no/partial/unknown) 必须有至少 1 条 evidence; verdict=unknown 是 OK 的 (诚实 > 编造)
- **D-E-04**: Audit doc 末尾"**Open Items 待 user 补**"段, 列出 unknown verdict 项, Phase 4.2 启动 ADR-V2-024 前 user 补足

### Phase 4.1 是否启动 ADR-V2-024 草稿 (gray area 6 — Option B, 4.1 出 Proposed)

- **D-F-01**: Phase 4.1 末 (audit doc 完成后) 启动 **ADR-V2-024 Proposed 草稿** (`docs/design/EAASP/adrs/ADR-V2-024-phase4-product-scope-decision.md`)
- **D-F-02**: 草稿用 `/adr:new --type strategy` 创建 (per ROADMAP §Phase 4.2 SC#2 措辞)
- **D-F-03**: ADR-V2-024 Proposed 草稿内容 = **Decision 段从 audit doc 三选一推荐填充** + **Rationale 段 §References 引用 audit doc** + **Consequences 段引用 §F Q1-Q4 答案** + **Enforcement 段空(留 Phase 4.2 填)**
- **D-F-04**: ADR-V2-024 状态机 = Phase 4.1 落盘 Proposed (F1-F3 lint pass), Phase 4.2 走 Proposed → Accepted (F4 lint pass after Decision text 锁定 + Enforcement 填完)
