## Gemini Review

# Phase 4.2 Implementation Plan Review — Decide & Document

This review evaluates the implementation plan for Phase 4.2 of the Grid project, focusing on decision-locking, ADR graduation, and milestone closure.

## 1. Summary
The Phase 4.2 plan is a robust, document-centric execution strategy designed to codify the outcomes of the Phase 4.1 audit. It successfully transitions the project from the "Leg A/B"二元 model to the "Engine vs. Data/Integration" double-axis model. The plan leverages empirical lessons from Phase 4.1, specifically regarding slash command limitations and cross-document fidelity. By combining mechanical terminology sweeps with high-stakes ADR state transitions and a final governance tracer-bullet (GOVERNANCE-03), the plan provides a clean exit for the Phase 4 milestone.

## 2. Strengths
*   **Audit-Fidelity Enforcement**: The use of the `audit-fidelity-grep` modality (T1-T3) ensures that the ADR reflects the audit's substance verbatim, eliminating manual re-interpretation errors.
*   **Operational Resilience**: T5 explicitly includes a "Path B" fallback for the `/adr:accept` slash command, proactively addressing the autonomous executor constraints discovered in Phase 4.1.
*   **Double-Axis Transition**: The terminology sweep (T6) is well-scoped, with clear exclusion lists (e.g., historical ADRs) and mandatory footnotes to prevent context loss during the "Leg A/B" deprecation.
*   **Governance Closure**: Integrating the deferred GOVERNANCE-03 (`/gsd-resume-work`) test into a mechanical sweep task is a strategic choice that minimizes cognitive load during context recovery.
*   **Strict Traceability**: T7 and T8 provide a comprehensive closure cascade, ensuring `PROJECT.md`, `ROADMAP.md`, and `REVIEW_POLICY.md` are synchronized with the final commit hashes.

## 3. Concerns
*   **Severity: LOW — Documentation Consistency**: T6 excludes the historical `ADR-V2-023` from the terminology sweep. While correct for ADR integrity, users reading the old strategy might find the "Leg A/B" terminology jarring without immediate cross-ref to the new axis model.
    *   *Mitigation*: The plan already includes a `superseded_by` link and a `deprecated_reason` field, which should suffice.
*   **Severity: LOW — Commit Atomicity in T7**: T7 combines three major files (`PROJECT.md`, `ROADMAP.md`, `REVIEW_POLICY.md`) into a single "close-phase cascade" commit. This diverges from the strict "one commit per task" rule.
    *   *Mitigation*: This is consistent with the Phase 4.0/4.1 precedents and is acceptable for administrative metadata updates.
*   **Severity: MEDIUM — GOVERNANCE-03 Failure Path**: The plan defines success criteria for the resume-work test but doesn't explicitly detail a recovery strategy if the checkpoint fails, other than "delegate to verifier."
    *   *Mitigation*: Ensure the verifier has a clear instruction to mark the requirement as `deferred` rather than `fail` to avoid blocking the milestone close if the gap is purely tool-based.

## 4. Suggestions
*   **Pre-Sweep Verification**: In T6, perform a final `git grep -i "Leg A"` across the entire workspace (including `examples/` and `sdk/`) just to identify if any "straggler" references exist outside the primary doc scope.
*   **F4 Warning Audit**: In T5, if the F4 lint produces warnings regarding module overlap (standard for strategy ADRs), capture the specific warning count in the commit message to differentiate it from a "clean" pass.
*   **Modality Table Polish**: In T7 (REVIEW_POLICY §3.5), ensure the "Executor inline acceptance" definition explicitly includes the "≥8 grep-assertion" requirement to maintain high standards for self-review.

## 5. Risk Assessment
**Overall Risk: LOW**

The plan involves zero changes to core engine logic, protocol buffers, or runtimes. The primary risks are documentation drift or linting errors, both of which are heavily mitigated by the 8-point automated acceptance criteria (grep/awk/lint) per task. The transition from "Leg A/B" to a double-axis model is purely structural/linguistic at this stage, making this a safe but critical "paperwork" phase before the next technical milestone.

## Codex Review

## Summary

The plan is directionally strong and unusually concrete: it maps DECIDE-02/03 into executable documentation tasks, captures the Phase 4.1 audit outputs, and treats ADR-V2-024 as the real milestone lock. The main risk is not strategic ambiguity; it is execution false-confidence. Several acceptance checks are internally inconsistent or brittle, and T6 in particular weakens "actually trigger `/gsd-resume-work`" into "create a trigger gate," which does not satisfy the stated GOVERNANCE-03 closure unless the user actually performs the resume test.

## Strengths

- The T1 layered ADR structure is the right shape: top-level dual-axis decision, then co-equal "双轴 substance" and "三选一推荐." This resolves the audit tension without discarding ROADMAP SC#1/SC#2.
- Open Items are captured with enough specificity: `0 家`, `60/30/10`, `grid-cli + grid-server`, `5+3`, and full supersede are all represented in tasks.
- T4 correctly treats ADR-V2-023 supersession as a bidirectional state transition, not just a note in ADR-V2-024.
- T5's Path B fallback is necessary and pragmatic given Phase 4.1 slash-command failures.
- The plan includes concrete grep/lint gates rather than relying on prose review.
- REVIEW_POLICY updates are a good closure loop from Phase 4.1 lessons into process.

## Concerns

### HIGH: T6 does not actually satisfy GOVERNANCE-03

The user decision says `/gsd-resume-work` should be triggered at T6 mid-sweep and judged on actual recovery. The plan downgrades this to creating `.continue-here.md`, then says autonomous executor cannot force `/clear`.

That is a useful checkpoint, but it is not the same requirement. If GOVERNANCE-03 is a milestone success criterion, this should either be a blocking manual checkpoint or be explicitly marked PARTIAL/deferred.

### HIGH: T6 sweep scope is inconsistent

The frontmatter/must-haves say in-scope files include:

- `CLAUDE.md`
- `.planning/PROJECT.md`
- `.planning/REQUIREMENTS.md`
- `.planning/ROADMAP.md`
- `docs/dev/WORK_LOG.md`
- `docs/design/EAASP/DEFERRED_LEDGER.md`

But Task 6 only sweeps three files:

- `CLAUDE.md`
- `WORK_LOG.md`
- `DEFERRED_LEDGER.md`

Then T7 separately edits `PROJECT.md` and `ROADMAP.md`, while `REQUIREMENTS.md` may be left with stale `Leg A/B` language. This can leave the milestone claiming "sweep complete" while active planning docs still contradict ADR-V2-024.

### HIGH: Several acceptance commands are likely wrong

Most notably T2 uses:

```bash
awk "/^### $opt:/,/^### Option /" $ADR
```

For `Option A`, the start line also matches the end regex, so many awk implementations will return only the heading line. The bullet/subheading checks may fail even when the document is correct.

This pattern repeats in several places. The plan should use section-specific end markers or stateful awk extraction.

### MEDIUM: T4 omits a required body update

The interface and must-haves require ADR-V2-023 body status line update:

```markdown
**Status:** Superseded by ADR-V2-024 (...)
```

But the T4 action focuses on frontmatter and does not explicitly perform this body edit. Since V2-023 is the first superseded ADR, a prominent body-level supersession notice is important.

### MEDIUM: T5 slash-command detection is not meaningful

`which adr` does not prove whether `/adr:accept` slash command is available. Slash commands are not shell binaries. The plan can still use Path B, but the detection should be removed or reframed as "autonomous mode assumes slash unavailable; use lint + manual frontmatter update."

### MEDIUM: Close-state updates are premature/self-referential

T7 marks ROADMAP success criteria and PROJECT validated state before T8 SUMMARY and verifier work. It also asks for the T7 commit hash inside the same T7 commit, which is impossible without amend/backfill.

Better: T7 should prepare close state, T8 should write SUMMARY, then a final close commit should mark final hashes and SCs.

### MEDIUM: Commit discipline is inconsistent

The plan says "8 atomic + 1 close-cascade = 9 commits," but T6 creates three subcommits, plus T1-T5, T7, T8, making roughly 10 commits. That is fine, but the plan should say so consistently and stop calling it strict one-task-one-commit.

### MEDIUM: `audit-fidelity-grep` validates presence, not judgment

For T1 verbatim copy, grep is enough. For T2 Alternatives and T7 policy changes, grep does not prove the reasoning is correct, only that expected phrases exist. Keep `audit-fidelity-grep`, but add a small semantic reviewer/verifier checklist for contradictions and stale framing.

### LOW: DEFERRED_LEDGER handling conflicts

The plan says `DEFERRED_LEDGER.md` has 0 occurrences/no-op, but later requires a see-link anchor in all three T6 files. If it truly has 0 occurrences, do not modify it just to satisfy an artificial anchor.

## Suggestions

- Make GOVERNANCE-03 explicit:
  - Option A: block after `.continue-here.md` and require user-run `/clear` + `/gsd-resume-work`.
  - Option B: mark GOVERNANCE-03 as PARTIAL and carry a deferred item.
  - Do not claim PASS from trigger-gate creation alone.

- Reconcile T6 scope:
  - Either sweep all active docs listed in must-haves, including `PROJECT.md`, `REQUIREMENTS.md`, and `ROADMAP.md`, or narrow the must-have text to the actual three files.
  - Add `ADR-V2-024` to the intentional allowlist, since it will necessarily contain historical `Leg A/B` terms.

- Fix awk section extraction before execution. Use marker-specific ranges, for example:
  - Option A: `awk '/^### Option A:/,/^### Option B:/'`
  - Option D: `awk '/^### Option D:/,/^## References/'`

- In T4, add explicit body-level supersession notice to ADR-V2-023 near the top, not only frontmatter.

- In T5, replace `which adr` with a clear fallback statement and run full lint after status update.

- Move final hash backfills and ROADMAP "complete" markings into a final close commit after T8/verifier, or avoid embedding self-hashes.

- Add one semantic verification pass after T1-T7:
  - no active doc frames Phase 4 as `Leg A/B` top-level
  - ADR-V2-024 Decision and Alternatives do not contradict Open Items
  - V2-023 is discoverably superseded
  - GOVERNANCE-03 status is truthful

## Risk Assessment

**Overall risk: MEDIUM.**

The strategic plan is sound, and the task breakdown is mostly complete. The risk comes from execution mechanics: inconsistent sweep scope, brittle verification commands, premature milestone closure, and the GOVERNANCE-03 trigger-gate not proving actual resume behavior. Fixing those would move the plan close to LOW risk.
